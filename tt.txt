게임엔진 제작에 필요한 프레임워크 구조


******* 게임 그래픽라이브러리 (API)의 인터페이스 구조 RHI (Render Hardware Interface) 로 구현 추상화가 핵심 (랜더링 할때 다양한 api를 사용가능하도록)
******* 로우 레벨의 자체 메모리 관리자를 직접 구현하도록 노력 9Custom Memory Manager)
******* 행렬 연산 시 위치, 회전값만 가지고 행렬르 계산하는 것에서 의문을 가짐, AI를 통해 더티 플레그 패턴과 그 적용 방법에 대해 확인
******* EnTT(ECS)가 무엇인지 이해한 상태에서 구조를 신경쓰며 엔진 제작
******* 엔티티 ID는 Index+version 2가지 형태가 합쳐져 있다. 하위20비트 인덱스, 상위 12비트 세대번호. 동일한 인덱스로 생성될 경우 구/신 을 세대번호로 구분
******* 'Render State' 관리 기법은 오버해드최소화 (잦은 상태변경으로 자원소모 최소화) 중복 상태 변경 방지, 상태정렬 최적화(여러 작업을 한번에), 상태오염 방지(꼬여서 잘못 렌더링 되는걸 방지) 를 위해 사용한다. PSO(Pipeline State Object) 라는 개념으로 사용하기도 한다(DX12, Vulkan)
******* 정적 라이브러리, 직접 포함 두가지 방식을 혼용해서 제작
******* 엔티티 메모리 재할당(Dangling Reference)  문제회피에 대해 공부 필요 (아마 배열할당이 꽉 차면 새로운 배열을 할당하고 이동하면서 주소가 바껴서 생기는 문제로 추측
******* fbx (마야나 블렌더)의 rootnode 이중변환 문제. IA가 문제를 제대로 이해하지만 사용자의 설명을 잘 이해하지 못해 여러번 같은 버그를 발생시킴. Assimp와 모델링의 구조를
분석해서 문제를 단계별로 나누고 하나씩 해결함
